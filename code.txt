# PROJECT ARCHITECTURE: UMHC TREASURER
# A financial management app for the University of Manchester Hillwalking Club.
# Stack: Vanilla JS (ES Modules) · Google Apps Script · Google Sheets · GitHub Pages

================================================================================
1. CORE PHILOSOPHY: SEPARATION OF CONCERNS
================================================================================

Strict unidirectional data flow:
  User Action -> Service Layer -> State (Store) -> UI Re-render

1. The UI is dumb. It renders data it receives and fires events. It does not
   fetch its own data or contain business logic.
2. The Store is the single source of truth. All application state lives here.
3. Services handle the dirty work: API calls, signing, Excel parsing, merging.

No npm / no build step. All dependencies are vendored locally.

    "Category":     string[],

4. CORE MODULES
================================================================================

A. State (src/core/state.js)
--------------------------------------------------------------------------------
Pub/Sub store. The single source of truth.

  store.subscribe(key, callback)   // returns unsubscribe fn
  store.setState(key, value)       // notify all subscribers for that key
  store.getState(key)              // returns deep clone (prevents mutation)

Safety:
  - Deep-clones all values in/out (structuredClone with custom fallback)
  - Deep equality check before notifying (no redundant renders)
  - Circular-reference detection in clone/equality
  - Blocks mutations to __proto__, constructor, prototype

B. DOM (src/core/dom.js)
--------------------------------------------------------------------------------
Declarative element creation. No innerHTML anywhere in the app (XSS prevention).

  el(tag, attrs, ...children)   // returns HTMLElement
    - attrs: className, style{}, dataset{}, onclick, onchange, …
  replace(element, ...newChildren)   // swap children, run cleanup on old ones
  cleanup(element)                   // recursively remove event listeners
  clear(element)                     // remove all children + cleanup

Event listeners are tracked in a WeakMap. cleanup() removes them all to prevent
memory leaks during re-renders.

C. Router (src/core/router.js)
--------------------------------------------------------------------------------
Hash-based. No page reloads.

  Routes: #dashboard · #transactions · #upload · #tags · #analysis · #settings

  - Listens on hashchange
  - Toggles CSS visibility of content sections
  - Updates active state in the sidebar
  - Redirects viewer-role users away from #upload

D. API Service (src/services/api.service.js)
--------------------------------------------------------------------------------
All backend calls go through here. Uses JSONP (Google Apps Script limitation).

Request Signing (HMAC-SHA256 via SubtleCrypto):
  - Login:    signed with the user's passkey
  - Subsequent calls: signed with sessionKey returned at login
  - Signature = HMAC-SHA256(secret, "action|timestamp|jsonParams")
  - Replay protection: server rejects timestamps older than ~5 minutes

Session Storage:
  - sessionId + sessionKey stored in sessionStorage (survives page reload,
    cleared on tab close)
  - Cleared automatically on 401 Unauthorized response

Request Deduplication:
  - Concurrent calls with identical params share one in-flight promise

Timeout: 15 seconds. JSONP script tags and callbacks cleaned up on
completion, error, or timeout.

Key methods:
  login(apiKey)                      // exchange passkey for session
  getAppData()                       // expenses + tags + balance + splits
  updateExpenses(data)               // save tag/transaction changes
  splitTransaction(original, splits) // create split group
  getSplitTransactions(page)         // paginated split history
  saveOpeningBalance(balance)

E. Auth Service (src/services/auth.service.js)
--------------------------------------------------------------------------------
Wraps ApiService for login/logout/session verification.

  AuthService.init()         // on app load — verify existing session
  AuthService.login(key)     // authenticate → set currentUser in store
  AuthService.logout()       // clear credentials + store
  AuthService.isLoggedIn()   // boolean check

Roles:
  "admin"  — full read/write access
  "viewer" — read-only; write API calls rejected server-side and
             mutation UI elements hidden client-side

F. Transaction Service (src/services/transaction.service.js)
--------------------------------------------------------------------------------
Merges split children into the main expense list after fetching from API.

  mergeSplits(rawExpenses, splitTransactions)
    - Removes SOURCE rows (the original unsplit transaction)
    - Injects CHILD rows in their place
    - Re-sorts by date

G. Financial Logic (src/core/financial.logic.js)
--------------------------------------------------------------------------------
Reconciles the opening balance with "Manual" transactions (historical entries
that predate the configured opening balance).

  Problem: Manual transactions represent activity already baked into the
  opening balance. Including them in totals would double-count.

  Solution:
    manualOffset           = manual_expenses - manual_income
    adjustedOpeningBalance = openingBalance + manualOffset
    currentBalance         = adjustedOpeningBalance + totalIncome - totalExpenses

  calculateFinancials(openingBalance, transactions) →
    { manualOffset, adjustedOpeningBalance, currentBalance, totalIncome, totalExpenses }

H. Excel Service (src/services/excel.service.js)
--------------------------------------------------------------------------------
Turns an uploaded .xlsx/.xls file into clean transaction JSON.

  - Scans first 20 rows for "date" and "description" headers
  - Locates Cash In / Cash Out columns by keyword matching
  - Normalises dates to YYYY-MM-DD (handles DD/MM/YYYY, ISO, datetime)
  - Combines multi-row transactions (same date, blank description)
  - Stops on footer-marker rows ("pending transactions", "please note…")

5. FEATURE COMPONENTS
================================================================================

A. Login (src/features/login/)
--------------------------------------------------------------------------------
- First-run Script URL setup (stored in localStorage)
- Passkey input + submit
- Session verification spinner (isVerifyingSession state)
- Error display

B. Dashboard (src/features/dashboard/)
--------------------------------------------------------------------------------
Displays:  Balance · Total Income · Total Expenses · Net Change
           Recent Transactions (table on desktop, cards on mobile)

Controls:  Timeframe selector (Current Month / 30 Days / 3 Months /
           6 Months / Year / All Time)

Uses calculateFinancials() for all balance figures.
Accessibility mode adds +/- symbols to currency values for colorblind users.

C. Transactions (src/features/transactions/)
--------------------------------------------------------------------------------
The most complex feature. Composed of several sub-modules:

  transactions.component.js     Main container + orchestration
  transactions.bulk.js          Bulk-select mode for applying tags to many rows
  transactions.filters.js       Category / Trip / Description filter bar
  transactions.logic.js         Pure functions: filterData(), sortData()

Capabilities:
  - Filter: Category, Trip/Event, Description search, "No Tag" magic value
  - Sort: Date, Description, Income, Expense, Net (type-aware)
  - Inline tag editing: click a tag pill to open TagSelector dropdown
  - Pending changes: edits batched client-side, saved in one API call
  - Bulk tag mode: select rows → apply tag to all at once
  - Split transaction: break one row into multiple with different categories
  - Manual transaction: add a historical entry (Type = "Manual")
  - Split history: view how a transaction was split
  - Mobile: cards grouped by MobileDisclosure sections

D. Tags (src/features/tags/)
--------------------------------------------------------------------------------
Manages the three tag taxonomies: Categories, Trips/Events, Types.

  TagsList      — overview with per-tag transaction counts
  TagsDetails   — drill into one tag, see its transactions
  TagsSubList   — secondary list (e.g. Types under a Trip)
  TagsAddTrip   — assign allowed Types to a Trip

Operations: add · rename · delete
Also manages:
  - TripTypeMap: which Types are allowed for each Trip
  - TripStatusMap: Active / Completed / Investment per Trip
  - Undo queue for destructive operations
  - Timeframe filter for transaction counts

E. Analysis (src/features/analysis/)
--------------------------------------------------------------------------------
Data visualisation powered by Chart.js.

Chart types:    Bar · Line · Pie · Doughnut
Metrics:        Balance · Income · Expenses · Net Income
X-axis grouping:Date (Daily/Weekly/Monthly/Yearly) · Category ·
                Trip · Trip Type · None (summary)
Filters:        Timeframe · Trip Status · specific Categories · specific Trips

Sub-components:
  AnalysisControls  — chart type + metric selectors
  AnalysisFilters   — category/trip/date pickers
  AnalysisChart     — Chart.js canvas wrapper
  AnalysisTable     — tabular export of chart data

Special rules enforced by the UI:
  - "Cumulative Balance" forces date grouping
  - Some chart types lock available metrics
  - Trip Type grouping only available when a Trip filter is active

F. Upload (src/features/upload/)
--------------------------------------------------------------------------------
Bulk import transactions from an Excel bank export.

  - Drag-and-drop or file picker
  - Uses ExcelService to parse
  - Duplicate detection against existing transactions
  - Preview table (toggle "Show New Only" to hide duplicates)
  - Uploads in batches of 20 rows to avoid GAS timeout
  - Admin-only route

G. Settings (src/features/settings/)
--------------------------------------------------------------------------------
  - Opening Balance: base starting balance (editable, saved to backend)
  - Manual Transaction Offset: auto-calculated, read-only
  - Effective Start Balance: adjustedOpeningBalance, read-only
  - Accessibility Mode: toggle colorblind-friendly +/- symbols (localStorage)

7. BACKEND (GOOGLE APPS SCRIPT)
================================================================================

Deployed as a publicly accessible Web App URL stored by the client in
localStorage. All requests must carry a valid HMAC-SHA256 signature.

Code.gs — Router (no business logic)
  doGet(e) → routes on e.parameter.action

  READ actions (admin + viewer):
    ping · getData · getAppData · getOpeningBalance ·
    getSplitGroup · getSplitHistory

  WRITE actions (admin only):
    saveData · updateExpenses · addTag · deleteTag · renameTag ·
    processTagOperations · saveOpeningBalance ·
    splitTransaction · revertSplit · editSplit

  AUTH actions:
    login — exchange passkey for sessionId + sessionKey + role

Config.gs — Server-side constants (passkey hash, sheet names, column indices)

Service_Auth.gs
  - Verifies HMAC-SHA256 signatures on every request
  - Validates timestamps (replay attack prevention)

Service_Session.gs
  - Generates random session tokens on login
  - Maps sessionId → { sessionKey, role, expiry }

Service_Sheet.gs — The ONLY file that touches SpreadsheetApp
  - Read/write rows in the "Finances" sheet
  - Opening balance in Config sheet cell C2
  - Column mapping: Date(A) Description(B) Income(C) Expense(D)
                    Category(E) Trip/Event(F) Type(G)
                    Split Group ID(H) Split Type(I)

Service_Tags.gs
  - Maintains Category, Trip/Event, Type lists
  - TripTypeMap and TripStatusMap stored in a dedicated Tags sheet
  - Bulk operation support (add/delete/rename in one call)

Service_Split.gs
  - Creates split groups: marks original as SOURCE, writes CHILD rows
  - Stores full split history in a Split History sheet
  - Supports revert (removes CHILD rows, un-marks SOURCE) and edit
  - Paginated history retrieval (500 rows per page)


================================================================================
8. SECURITY MODEL
================================================================================

Frontend:
  - No innerHTML anywhere (XSS prevention)
  - HMAC-SHA256 request signing (SubtleCrypto)
  - Session credentials in sessionStorage (cleared on tab close)
  - Input sanitisation: escapeHtml(), sanitizeForId() in utils.js

Backend:
  - Every request signature-verified before processing
  - Role check before every write action
  - Passkey never stored in plaintext (hashed in Config.gs)
  - Timestamp validation window prevents replay attacks

Known tradeoff:
  JSONP is used instead of fetch due to Google Apps Script CORS constraints.
  This is less ideal than a proper REST endpoint but is mitigated by:
  strict URL control, unique per-request callback names, immediate cleanup.

10. DATA FLOW: WORKED EXAMPLE — "Splitting a Transaction"
================================================================================

1. User clicks a transaction row in the Transactions table.
2. TransactionsComponent opens SplitTransactionModal.
3. User defines splits: £60 → Category A, £40 → Category B.
4. Modal calls ApiService.splitTransaction(original, splits).
5. ApiService signs the request and fires a JSONP call.
6. Backend (Code.gs) routes to Service_Split.gs:
     - Marks original row as Split Type = "SOURCE"
     - Appends two CHILD rows for the splits
     - Records everything in the Split History sheet
7. Backend returns success.
8. ApiService calls store.setState("rawExpenses", updatedData) and
   invalidates the splitTransactions cache.
9. TransactionService.mergeSplits() re-processes:
     SOURCE row suppressed, CHILD rows injected in its place.
10. store.setState("expenses", mergedData) fires.
11. TransactionsComponent subscriber re-renders the table automatically.
    The original row is gone; the two split rows appear in its place.

11. HOW TO ADD A NEW FEATURE
================================================================================

Example: "Budget Goals"

1. Data shape: { category, amount, month }
2. Backend: Service_Budget.gs — read/write a Budgets sheet tab.
   Register actions in Code.gs: getBudgets, saveBudget, deleteBudget.
3. API methods: add getBudgets(), saveBudget(), deleteBudget() to api.service.js.
4. Store: add budgets: [] to the store default state.
5. Component: src/features/budgets/budgets.component.js
     - subscribe to "budgets" in the store
     - render using el() — no innerHTML
     - on user action, call api.service.js methods
     - on success, call store.setState("budgets", newData)
6. Router: register "#budgets" in router.js.
7. Nav: add a sidebar link in index.html.

No existing feature files need to be touched.

12. DEPLOYMENT
================================================================================

Frontend:
  Push to main → GitHub Actions deploys to GitHub Pages automatically.
  No build step. Files served as-is.

Backend:
  1. Paste .gs files into a Google Apps Script project.
  2. Set passkey hash and sheet ID in Config.gs.
  3. Deploy as Web App (Execute as: Me, Access: Anyone).
  4. Copy the Web App URL.
  5. Paste URL into the app on first login (stored in localStorage).
